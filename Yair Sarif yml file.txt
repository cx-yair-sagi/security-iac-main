Question: Why does uploading Checkmarx SARIF to GitHub Code Scanning fail with locationFromSarifResult: expected at least one location, and is this related to AST-148152?
Analysis date: June 25, 2026

Direct Answer
Yes — your GitHub upload failure and AST-148152 are the same root cause: the Checkmarx SAST SARIF exporter places the attack-vector nodes (the code-flow path between source and sink) inside the top-level result.locations array instead of inside result.codeFlows[].threadFlows[].locations[]. GitHub's SARIF ingestion engine requires every result to have a valid, resolvable top-level locations entry with an artifactLocation (i.e., a real file URI). When the SAST exporter writes flow nodes there instead, GitHub either sees entries that lack a proper artifactLocation URI, or sees an empty top-level locations array, and rejects the entire file with locationFromSarifResult: expected artifact location / expected at least one location. This is a known, confirmed bug in the SAST SARIF report generator, tracked as AST-146432 — which was the canonical bug that AST-148152 was marked a duplicate of. A fix was planned for CLI 2.3.52 (release date June 5, 2026) — check whether that version is available to your customers.

Detailed Analysis
Entry point

The SARIF is produced by the CxOne CLI / SAST report generator when --report-format sarif is passed. The internal mapping is documented in the internal Confluence page SAST → SARIF Mapping.

The structural bug — what the SAST exporter does vs. what it should do

The SARIF 2.1.0 spec defines two distinct fields on a result object:

Field	SARIF meaning	Cardinality
result.locations[]	Where the vulnerability manifests — the single "sink" location GitHub displays as the alert location	REQUIRED — must contain ≥1 entry with a valid artifactLocation.uri
result.codeFlows[].threadFlows[].locations[]	The attack vector / data-flow path from source to sink	OPTIONAL
What SAST actually emits (confirmed by the Confluence mapping table): all SAST Nodes (the entire flow path, not just the sink) are mapped to codeFlows.threadFlows.locations[]. That part is correct per the spec.

The problem is the result.locations field is absent or empty. GitHub's parser requires at minimum one entry in result.locations pointing to the sink node (the final node in the attack vector — where the vulnerability is reachable). Without that, every result fails the locationFromSarifResult check.

This is precisely what AST-112622 (the Salesforce case with the identical error message) and AST-146432 documented.

Why your hypothesis is correct but slightly imprecise

You wrote: "GitHub now enforces that all locations are referenced at the top level, where locations listed under codeFlows/threadFlows are not valid." That's accurate in effect but the mechanism is: GitHub doesn't reject codeFlows locations — it simply also requires at least one result.locations[] entry with a valid artifactLocation. The current SAST export omits that top-level entry entirely, so every result fails the check.

Data flow

SAST scan results (internal JSON)
  → CLI SARIF reporter
    → Maps each Result.Nodes[] → codeFlows.threadFlows.locations  ✅ correct
    → Does NOT populate result.locations[]                          ❌ missing
  → cx_result.sarif written to disk
    → github/codeql-action/upload-sarif@v3
      → GitHub SARIF ingestor: locationFromSarifResult check fails
        → Upload rejected
What the fix should look like (plain terms)

The SARIF reporter needs to additionally populate result.locations[] with a single entry pointing to the sink node — specifically the last Node in the attack-vector path. That node already has FileName, Line, Column, and Length available. Populating result.locations[0].physicalLocation.artifactLocation.uri with that node's file path, and region.startLine/startColumn with that node's position, is all GitHub needs.
